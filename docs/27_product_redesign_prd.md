# BeautyLens Product Redesign PRD

작성일: 2026-06-30

## 1. 현재 문제 요약

BeautyLens는 현재 추천, 상품 탐색, 상품 상세, 댓글, 찜, 내부 평가, 추천 피드백, 최근 본 상품, 얼굴 로그인, 관리자 상품 운영, 신고 관리, 활동 로그까지 기능 범위는 넓다. 그러나 기능이 화면 흐름 안에 자연스럽게 배치되기보다 “추가된 기능 목록”처럼 노출되어 실제 서비스처럼 보이지 않는 문제가 있다.

핵심 문제는 다음과 같다.

- 상품 카드에 행동 버튼이 많아 사용자가 가장 중요한 행동인 “상품 이해”와 “상세 보기”에 집중하기 어렵다.
- 추천 카드에서 좋아요, 별로예요, 관심 없음이 직접 노출되어 추천 서비스보다 설문 UI처럼 보인다.
- 사용자가 상품 상세 정보를 충분히 보기 전에 평가를 남길 수 있어 내부 평가 데이터 품질이 낮아질 수 있다.
- 사이트 사용자 평가와 회원 의견이 같은 의견 입력처럼 보이며, 각각의 목적이 분리되지 않는다.
- 마이페이지가 세로로 길고 정보 구조가 없어 활동 관리 화면이라기보다 데이터 목록처럼 보인다.
- 얼굴 로그인 설정에 `insightface-buffalo_l` 같은 기술 모델명이 노출되어 사용자 신뢰를 떨어뜨린다.
- 얼굴 등록/얼굴 로그인 과정이 단계형 인증 경험이 아니라 단순 카메라 기능처럼 보인다.
- 관리자 화면이 운영 업무 흐름보다 개별 기능 페이지의 나열에 가깝다.
- 상품 탐색/추천/상세의 연결이 커머스 또는 추천 서비스의 자연스러운 흐름으로 이어지지 않는다.

따라서 이번 재설계의 목표는 기능을 더 추가하는 것이 아니라, 이미 구현된 기능을 사용자의 의사결정 흐름과 운영자의 관리 흐름에 맞게 재배치하는 것이다.

## 2. 제품 정체성

BeautyLens는 피부 타입과 리뷰 데이터를 기반으로 사용자가 화장품을 더 신중하게 선택하도록 돕는 추천 서비스다.

조금 더 구체적으로 정의하면:

BeautyLens는 크롤링 리뷰 분석과 BeautyLens 내부 사용자 반응을 함께 활용해, 피부 타입별 화장품 선택의 근거를 보여주는 리뷰 기반 추천 서비스다.

서비스가 강조해야 할 가치는 다음과 같다.

- 단순 인기 상품이 아니라 피부 타입과 리뷰 근거를 함께 본다.
- 외부 크롤링 리뷰와 내부 사용자 평가를 분리해 신뢰도를 높인다.
- 사용자는 상품을 충분히 본 뒤 평가를 남긴다.
- 관리자는 상품과 추천 품질을 운영 지표로 관리한다.

## 3. 핵심 사용자 흐름

### 첫 방문자

목표: BeautyLens가 무엇을 하는 서비스인지 빠르게 이해하고, 추천 또는 상품 탐색으로 진입한다.

흐름:

1. 홈에서 “피부 타입과 리뷰 데이터를 기반으로 화장품 선택을 돕는다”는 한 줄 설명을 본다.
2. 추천 받기 또는 상품 둘러보기 CTA를 선택한다.
3. 피부 타입별 추천 진입 카드로 빠르게 추천 페이지로 이동한다.
4. 상품 카드에서 이미지, 브랜드, 추천 점수, 간단한 추천 이유를 본다.
5. 자세히 보기를 통해 상품 상세에서 리뷰 근거를 확인한다.

첫 방문자에게 숨길 것:

- 관리자 CTA
- 얼굴 로그인 강조
- 기술 스택 문구
- 내부 모델명
- 시연용 문구

### 로그인 회원

목표: 개인 피부 타입을 설정하고, 추천/찜/평가/최근 본 상품을 계정 단위로 관리한다.

흐름:

1. 로그인 후 내 피부 타입과 피부 고민을 기준으로 추천을 받는다.
2. 상품 카드에서는 찜 또는 상세 보기만 수행한다.
3. 상세에서 리뷰 근거와 내부 평가를 확인한다.
4. 상품을 충분히 본 뒤 별점, 자극 여부, 재구매 의사를 남긴다.
5. 마이페이지에서 찜, 평가, 최근 본 상품, 댓글, 추천 피드백을 탭으로 관리한다.

### 추천을 받는 사용자

목표: 나에게 맞는 상품 후보를 빠르게 비교하고, 관심 없는 추천은 조용히 제외한다.

흐름:

1. 추천 페이지에서 피부 타입을 확인한다.
2. 필요하면 피부 고민과 플랫폼 필터를 조정한다.
3. 추천 기준 설명을 읽고 결과를 본다.
4. 카드에서는 상세 보기와 찜만 주 행동으로 사용한다.
5. 추천이 맞지 않을 경우 “이 추천 숨기기” 또는 보조 메뉴에서 피드백을 남긴다.
6. 깊은 평가는 상품 상세에서 한다.

### 상품을 탐색하는 사용자

목표: 브랜드, 플랫폼, 이미지 여부, 피부 타입 기준으로 상품을 찾는다.

흐름:

1. 상품 탐색에서 검색어를 입력한다.
2. 플랫폼, 피부 타입, 이미지 있는 상품 필터를 선택한다.
3. 정렬은 추천 점수, 리뷰 수, 평점 중심으로 둔다.
4. 상품 카드에서 핵심 정보만 보고 상세로 이동한다.

### 상품 상세를 보는 사용자

목표: 상품 추천의 이유와 주의 신호를 이해하고 구매 판단을 보조받는다.

흐름:

1. 상품 요약에서 이미지, 브랜드, 상품명, 플랫폼, 추천 점수를 본다.
2. 추천 근거에서 왜 추천됐는지 확인한다.
3. 리뷰 기반 분석에서 긍정 리뷰와 주의 리뷰를 분리해 본다.
4. 사이트 사용자 평가에서 내부 별점, 자극 여부, 재구매 의사 요약을 본다.
5. 충분히 본 뒤 내 평가를 남긴다.
6. 회원 의견은 자유 댓글로 별도 확인한다.

### 평가를 남기는 사용자

목표: 구매 또는 사용 경험에 가까운 신호를 남긴다.

흐름:

1. 상품 상세에서 리뷰와 사용자 평가를 확인한다.
2. “내 평가 남기기” 영역에서 별점, 자극 여부, 재구매 의사를 입력한다.
3. 짧은 평가 메모를 선택적으로 남긴다.
4. 자유로운 대화성 의견은 “회원 의견”에 댓글로 남긴다.

평가 UX 원칙:

- 상품 카드에서 바로 깊은 평가를 받지 않는다.
- 별점은 상품 상세에서만 입력한다.
- 댓글과 평가는 목적을 분리한다.

### 마이페이지에서 활동을 관리하는 사용자

목표: 내 정보와 활동을 탭/사이드바로 관리한다.

흐름:

1. 마이페이지에 들어오면 기본 탭은 내 프로필이다.
2. 좌측 사이드바에서 피부 타입, 얼굴 로그인, 찜한 상품, 내가 평가한 상품, 최근 본 상품, 내가 남긴 의견, 추천 피드백을 선택한다.
3. 오른쪽에는 선택한 섹션만 표시된다.
4. 사용자는 찜 해제, 평가 수정, 댓글 삭제, 추천 피드백 취소를 해당 섹션에서 처리한다.

### 얼굴 로그인을 설정하는 사용자

목표: 기술을 의식하지 않고 안전하고 쉬운 생체 로그인 설정을 완료한다.

흐름:

1. 마이페이지의 얼굴 로그인 탭으로 이동한다.
2. 등록됨/미등록 상태와 최근 등록일을 본다.
3. 얼굴 등록 또는 재등록을 누른다.
4. 단계형 wizard에서 안내를 따라 정면, 왼쪽, 오른쪽, 위쪽, 아래쪽을 촬영한다.
5. 등록 완료 피드백을 받는다.
6. 모델명, 벡터, 임베딩 같은 기술 용어는 보지 않는다.

### 관리자

목표: 서비스 운영 상태를 대시보드에서 확인하고, 문제 상품/댓글/추천 품질을 처리한다.

흐름:

1. 관리자 대시보드에서 전체 상품 수, 숨김 상품 수, 이미지 없는 상품 수, 신고 대기 수, 내부 평가 수, 최근 운영 로그를 확인한다.
2. 상품 관리에서 검색/필터로 상품을 찾고 숨김, 추천 제외, 품질 상태, 운영 메모를 관리한다.
3. 댓글/신고 관리에서 신고 사유를 보고 처리하거나 댓글을 복구한다.
4. 추천 품질 관리에서 추천 점수는 높은데 부정 신호가 높은 상품, 이미지 없는 추천 상품, 관심 없음이 많은 상품을 확인한다.
5. 운영 로그에서 관리자 작업 내역을 추적한다.

### 찜

- 가벼운 관심 표시
- 상품 카드와 상세에서 가능
- 추천 품질의 약한 관심 신호

### 별점

- 상품 상세에서만 입력
- 상품 정보를 본 뒤 남기도록 유도
- 내부 만족도 지표로 사용

### 자극 여부

- 상품 상세 평가 영역에서 입력
- 피부 타입별 민감도 분석에 사용

### 재구매 의사

- 상품 상세 평가 영역에서 입력
- 만족도의 강한 신호로 사용

### 추천 피드백

- 추천 페이지에서 바로 깊은 평가를 받지 않는다.
- “이 추천을 다시 보고 싶지 않아요” 정도로 가볍게 둔다.
- 좋아요/별로예요는 상품 상세 또는 추천 결과 피드백 모달에서 받는다.

### 최근 본 상품

- 사용자가 실제로 상세를 본 상품만 기록한다.
- 단순 카드 노출은 최근 본 상품으로 기록하지 않는다.

## 14. 추천 점수/서비스 점수 설계

기존 `recommendationScore`는 크롤링 리뷰 기반 점수로 유지한다.

`serviceScore`는 BeautyLens 내부 사용자 반응 기반 보조 점수로 둔다.

### 구분

- 추천 점수: 외부 리뷰/전처리 기반
- 서비스 점수: BeautyLens 내부 사용자 반응 기반

### 화면 표시 원칙

추천 카드:

- 기본은 추천 점수 중심
- 서비스 점수는 보조 배지 또는 숨김 가능

상품 상세:

- 추천 점수와 서비스 반응을 분리 표시
- 내부 평가가 부족하면 부족하다고 표시

### 데이터 부족 문구

- 아직 내부 평가가 부족합니다.
- 더 많은 사용자 평가가 쌓이면 추천 품질에 반영됩니다.

## 19. 전역 UX 품질 기준

### 19.1 사용자 화면 원칙

BeautyLens의 모든 사용자 화면은 다음 원칙을 따른다.

- 사용자가 지금 무엇을 해야 하는지 바로 알 수 있어야 한다.
- 한 화면에 같은 목적의 입력창이 반복되면 안 된다.
- 버튼은 주 행동과 보조 행동이 구분되어야 한다.
- 사용자가 모르는 내부 코드, DB 값, API 값, 모델명은 보이지 않아야 한다.
- 데이터가 없을 때 빈 배열, `null`, `undefined` 같은 개발자식 값이 보이면 실패다.
- 로그인 후 원래 하려던 행동으로 돌아와야 한다.
- 오류가 발생해도 전체 화면이 죽지 않고 해당 섹션만 실패 처리되어야 한다.
- 실제 서비스라면 이 문구와 배치를 그대로 써도 어색하지 않아야 한다.

### 19.2 사용자 화면에 보이면 실패인 값

다음 값이 사용자 화면에 그대로 노출되면 실패로 본다.

- `[]`
- `{}`
- `null`
- `undefined`
- `NaN`
- `Y`
- `N`
- `ACTIVE`
- `NORMAL`
- `DELETED`
- `PENDING`
- `SPAM`
- `ETC`
- `FALSE_INFO`
- 내부 enum 코드
- DB ID
- API 경로
- JavaScript 에러 메시지 원문
- 서버 예외 메시지 원문
- 기술 모델명
- 개발자용 변수명

단, 서버 내부 코드, 로그, 주석에만 존재하고 사용자 화면에 출력되지 않는 경우는 실패로 보지 않는다.

### 19.3 빈 상태 처리 원칙

데이터가 없을 때는 화면을 비워두거나 `[]`를 보여주지 않는다.

권장 문구:

- 아직 긍정 리뷰가 충분하지 않습니다.
- 아직 주의 리뷰가 충분하지 않습니다.
- 아직 회원 의견이 없습니다.
- 아직 찜한 상품이 없습니다.
- 최근 본 상품이 없습니다.
- 아직 사이트 사용자 평가가 부족합니다.
- 평가가 쌓이면 별점, 자극 여부, 재구매 의사 요약에 반영됩니다.
- 표시할 운영 로그가 없습니다.
- 신고된 댓글이 없습니다.
- 조건에 맞는 상품이 없습니다.

### 19.4 API 실패 처리 원칙

보조 API 하나가 실패해도 전체 화면이 죽으면 안 된다.

예시:

- 긍정 리뷰 API 실패: 긍정 리뷰 섹션만 실패 문구 표시
- 주의 리뷰 API 실패: 주의 리뷰 섹션만 실패 문구 표시
- 댓글 API 실패: 회원 의견 섹션만 실패 문구 표시
- 최근 본 상품 기록 실패: 사용자 화면은 유지하고 내부 로그만 남김
- 내 평가 조회 실패: 평가 입력 영역은 표시하되 기존 평가를 불러오지 못했다는 문구 표시

사용자 문구 예시:

- 리뷰를 불러오지 못했습니다. 잠시 후 다시 확인해 주세요.
- 회원 의견을 불러오지 못했습니다. 잠시 후 다시 확인해 주세요.
- 평가 정보를 불러오지 못했습니다. 다시 시도해 주세요.
- 일시적으로 저장하지 못했습니다. 잠시 후 다시 시도해 주세요.

### 19.5 로그인 후 복귀 원칙

사용자가 특정 화면에서 로그인이 필요한 행동을 하려 했다면, 로그인 성공 후 원래 화면으로 돌아와야 한다.

예시:

- 상품 상세에서 평가하려고 로그인: 로그인 성공 후 해당 상품 상세로 복귀
- 상품 상세에서 댓글 작성하려고 로그인: 로그인 성공 후 해당 상품 상세로 복귀
- 상품 상세에서 찜하려고 로그인: 로그인 성공 후 해당 상품 상세로 복귀
- 마이페이지 접근 중 로그인: 로그인 성공 후 마이페이지로 복귀
- 추천 페이지에서 로그인: 로그인 성공 후 추천 페이지로 복귀

구현 조건:

- 로그인 링크는 `redirect` 값을 포함한다.
- redirect는 내부 경로만 허용한다.
- 외부 URL, `javascript:`, `//외부도메인`, 제어 문자가 포함된 값은 차단한다.
- 잘못된 redirect 값은 기본 홈 또는 기존 로그인 성공 경로로 보낸다.

### 19.6 정량 평가와 회원 의견 구분 원칙

정량 평가와 회원 의견은 목적이 다르다.

정량 평가:

- 별점
- 자극 여부
- 재구매 의사
- 짧은 평가 메모
- BeautyLens 내부 만족도 요약에 사용

회원 의견:

- 자유 댓글
- 사용 후기
- 질문
- 다른 회원에게 보이는 의견

화면 원칙:

- 두 개의 큰 textarea가 연속으로 나오면 중복 입력처럼 보이므로 피한다.
- `내 평가 남기기`는 compact한 평가 폼으로 구성한다.
- `회원 의견`은 댓글 목록 중심으로 구성한다.
- 평가 저장 버튼과 의견 작성 버튼은 목적이 구분되어야 한다.
- 설명 문구로 두 영역의 차이를 분명히 알려준다.

### 19.7 관리자 화면 UX 원칙

관리자 화면도 단순히 API가 동작하는지만 보면 안 된다.

관리자 화면은 다음 기준을 따른다.

- 지금 처리해야 할 항목이 먼저 보여야 한다.
- 상품 상태, 추천 상태, 신고 상태, 품질 상태가 문구로 이해되어야 한다.
- 원시 enum 값이 그대로 보이면 안 된다.
- 필터와 검색이 운영 흐름에 맞게 배치되어야 한다.
- 위험한 작업은 확인 절차 또는 명확한 버튼 문구가 있어야 한다.
- 운영 메모와 활동 로그는 관리자가 왜 변경했는지 추적할 수 있어야 한다.

### 19.8 모든 Phase 공통 자체 검수 질문

각 Phase 완료 후 Codex는 아래 질문에 답해야 한다.

- 사용자가 이 화면에서 무엇을 해야 하는지 바로 알 수 있는가?
- 주 행동과 보조 행동이 구분되는가?
- 같은 의미의 입력창이 반복되어 보이지 않는가?
- 데이터가 없을 때 자연스러운 빈 상태가 보이는가?
- 사용자 화면에 `[]`, `null`, `undefined`, 원시 enum 값이 보이지 않는가?
- 로그인 후 원래 하려던 행동으로 돌아올 수 있는가?
- API 일부 실패 시 전체 화면이 죽지 않는가?
- 문구가 실제 서비스에서 그대로 써도 어색하지 않은가?
- 모바일에서 버튼과 입력창이 과하게 길거나 깨지지 않는가?
- 관리자 화면은 실제 운영 흐름에 맞는가?
- 기능은 살아 있지만 위치가 부자연스럽지는 않은가?
- 이 기능이 사용자에게 꼭 필요한 위치에 있는가?
- 숨겨야 할 기능이 너무 앞에 나오고 있지는 않은가?

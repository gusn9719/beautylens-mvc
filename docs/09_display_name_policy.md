작성일: 2026-06-30

## 1. 배경 및 기본 방침

### 왜 DB의 PRODUCT_NAME을 수정하지 않는가

| 이유 | 설명 |
|---|---|
| 원본 보존 | oliveyoung/musinsa의 실제 판매 상품명이 원본. 추후 참조/링크 생성에 필요. |
| 롤백 불가 | DB UPDATE 후 되돌리려면 원본 백업이 필요하고 별도 테이블/컬럼 추가가 수반됨. |
| 책임 범위 | 화면 표시는 UI/Service 계층 책임이며 DB는 데이터를 저장하는 계층. |
| 스크래핑 일치 | product_key 기준으로 원본 플랫폼 URL과 매핑되므로 이름 변경이 일관성을 깨뜨릴 수 있음. |

### 왜 displayName을 별도로 두는가

- 추천/상품 API 응답에서 `productName` (원본)과 `displayName` (정제본) 두 필드를 노출
- 프론트엔드는 기본적으로 `displayName` 사용, 필요 시 `productName` 원본 참조
- 정제 로직이 잘못된 경우 Service 계층 코드만 수정하면 되므로 안전

## 2. 정제 규칙 (v1.0)

### Rule 1: 앞 대괄호 제거

상품명 맨 앞의 대괄호 `[...]` 블록이 프로모션 키워드를 포함하면 제거 후보.

**제거 금지 우선 조건** (이 조건이 있으면 무조건 유지):
- 대괄호 안에 수량/용량 정보가 있는 경우: `2PACK`, `2pack`, `10매`, `100ml` 등
  → `[2PACK]`, `[10매기획]` 등은 제품 구성 정보이므로 유지

**Safe 제거 (자동 제거)** — 대괄호 내용이 다음을 포함하면 제거:
```
올영픽, 하루특가, 타임어택, 무신사 단독, 사은품, 포켓몬, 콜라보,
런칭, 에디션, 한정판, 시간한정, 올영, 올리브영픽, 위클리픽, 오늘의픽
```

**Review 제거 (조건부 제거 — 이번 단계에서는 보류)** — 대괄호 내용이 다음을 포함:
```
PICK, NEW, 단독, 기획, 증정, 1+1, 세일, 특가, 한정, 할인, 이벤트, 프로모션, 쿠폰, 무료배송
```

> **[SET] 관련 판정**: 사용자 지침에 "세트는 제품 구성일 수 있으므로 무조건 제거하지 말 것" 명시.
> dry-run 결과에서도 `[SET]` 제거 항목 대부분이 review 등급. → **[SET]은 review 대상에서 제외, 유지 처리**.

**처리 방식**:
- 앞 대괄호가 safe 제거 대상이면 제거하고 다음 대괄호 확인 반복
- 수량/용량 정보가 있는 대괄호에서 중단
- Review 대상은 이번 단계에서 API 미반영, 별도 검토

### Rule 2: 뒤쪽 증정품 문구 제거

상품명 뒤쪽의 증정품 관련 문구 제거.

**패턴 A**: `+(증정) xxx` 형식 (공백 포함 끝까지):
```
바디로션 300ml (3종 택1) +(증정) 고체향수키링 레몬비누향
→ 바디로션 300ml (3종 택1)
```

**패턴 B**: `(+xxx)` 형식에서 증정품 키워드 포함 시:
```
AHC 아이크림 30ml 기획 (+세럼 10ml+피카츄 액정클리너)
→ AHC 아이크림 30ml 기획
```
증정품 키워드: `키링, 스티커, 파우치, 거울, 타올, 케이스, 아크릴, 포스트잇, 마켓백, 브러쉬, 굿즈, 미니어처, 액정클리너, 머리핀, 헤어핀, 클리너, 틴케이스, 이어폰, 볼펜, 부채, 에코백, 집게`

**제거하지 않는 것**:
- `(+50ml)`, `(+리필팩)` 처럼 볼륨 정보만 있고 증정품 키워드 없는 경우: 제품 구성으로 유지
- `(3종 택1)`, `(단품/기획)` 등 선택 옵션 표기: 유지

### Rule 3: 정제 후 안전 확인

- 정제 후 8자 미만 → 원본 유지, riskLevel=unsafe
- 볼륨 정보가 원본에 있었는데 정제 후 모두 사라진 경우 → riskLevel=review

### Rule 4: 공백 정규화

연속 공백(2개 이상) → 단일 공백으로 치환.

```
에이시카 365 수분토너 pH4.5  200ml → 에이시카 365 수분토너 pH4.5 200ml
```

## 7. 최종 정책 판정

### 판정: **B안**

> "정제 규칙이 일부 위험하다. API 반영 전에 review/unsafe 케이스를 더 줄여야 한다."

**판정 근거**:

| 구분 | 건수 | 판단 |
|---|---|---|
| safe (앞 대괄호 안전 제거 + 공백 정규화) | 162건 | ✅ API 반영 가능 |
| review 중 [증정], [PICK], [유명인명 PICK] 제거 | 다수 | ✅ 결과가 합리적 |
| review 중 **[SET] 제거** | 다수 | ❌ 사용자 지침 위반 가능. 제거 금지로 변경 필요 |
| review 중 gift suffix 제거 | 46건 | ⚠️ 볼륨 정보 소실 가능성 있는 케이스 존재 |
| unsafe (원본 유지) | 3건 | ✅ 원본 보존, 문제 없음 |
| 미포착 패턴 4종 | — | ⚠️ 다음 버전에서 보완 필요 |

**A안을 선택하지 않은 이유**:
- `[SET]` 처리 방침이 사용자 지침(`세트는 제품 구성일 수 있으므로 제거하지 말 것`)과 충돌
- `(증정: xxx)`, `+gift_word` 등 미포착 패턴이 존재해 일관성 부족

**C안을 선택하지 않은 이유**:
- safe 162건은 이미 안전한 결과를 생성함
- displayName 자동 정제가 불가능한 수준은 아님

## 8. 다음 단계 권장 사항

### A. 규칙 보완 후 재실행 (v1.1)

1. `[SET]` → review 제거 목록에서 제외 (보존)
2. `+(증정)` 이외의 뒤쪽 증정 패턴 추가
3. `(증정: xxx)` 형식 추가
4. 보완된 규칙으로 `propose_display_names.py` 재실행

### B. safe 162건 먼저 API 반영

규칙 보완 전이라도 safe(162건)는 API 반영 가능.

**반영 방법** (별도 단계에서 진행):
1. `ProductVO` + `RecommendationVO`에 `displayName` 필드 추가 (String)
2. `ProductService`/`RecommendationService` 계층에서 `productName` → `displayName` 생성 메서드 호출
3. 정제 로직은 Java `DisplayNameCleaner` 유틸리티 클래스로 구현
4. DB는 수정하지 않음 — Service가 응답 시 on-the-fly 계산

### C. review 건 별도 검토

review 282건은 `docs/display_name_candidates.csv`에서 `riskLevel=review` 필터로 확인.  
특히 `[SET]` 항목은 보존, `[증정]`/`[PICK]` 항목은 제거 가능으로 분류 추천.

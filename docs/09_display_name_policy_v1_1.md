## 9. 최종 판정

### 판정: **B안**

> safe 161건은 API 반영 가능. review 305건은 원본 유지(productName 그대로 반환).

**판정 근거**

| 분류 | 건수 | 처리 권장 |
|---|---|---|
| safe | 161 | ✅ Java API에서 on-the-fly 자동 적용 |
| review | 305 | 🔶 원본 `productName` 그대로 반환 (displayName = productName) |
| unsafe (원본 유지) | 3 | ✅ 원본 유지 (이미 처리됨) |
| NO_CHANGE | 1,055 | ✅ 그대로 |

**A안을 선택하지 않은 이유**:
- review 305건 중 일부(`[화해1위/쿨링진정]` 형식)는 제품 특성 설명과 마케팅 클레임이 혼재하여  
  자동 전체 적용 시 정보 소실 케이스 존재.
- `[아떼X세터]`, `(세터 콜라보 거울 키링 & ...)` 등 소수 미포착 패턴 있음.
- review 전체를 신뢰하기 위한 수동 검증 부담.

**C안을 선택하지 않은 이유**:
- safe 161건은 명확히 안전하며, displayName 자동 정제가 의미 있음.
- 특히 [포켓몬 에디션], [타임어택], [올영픽] 등 명백한 프로모션 제거는 가치 높음.

**B안 구체 적용 방법**:
- API가 displayName을 반환할 때: `riskLevel=safe`이면 정제된 이름, 그 외이면 원본 productName
- review 305건은 별도 사람 검수 후 v1.2에서 추가 반영 가능
- 현재 도구: `display_name_candidates_v1_1.csv`에서 `riskLevel=safe` 필터로 확인 가능

## 10. Java API 반영 계획 (다음 단계)

다음 단계에서 진행할 내용 (아직 하지 않음):

1. **`DisplayNameCleaner.java`** 유틸리티 클래스 작성
   - Python 규칙을 Java로 포팅
   - `clean(String productName) → String displayName` 메서드
   - `riskLevel` 반환 포함

2. **`ProductVO` + `RecommendationVO`에 `displayName` 필드 추가**
   - DB는 수정하지 않음
   - Service 계층에서 `productName` → `DisplayNameCleaner.clean()` 호출

3. **반영 범위 확정**
   - safe: 자동 적용
   - review/unsafe: `displayName = productName` (원본 그대로)

4. **Mapper/VO 수정 범위**
   - `ProductVO.java`: `displayName` 필드 추가
   - `RecommendationVO.java`: `displayName` 필드 추가
   - Mapper XML: 수정 없음 (DB에서 가져오는 것 아님)
   - Service: `DisplayNameCleaner` 호출 추가
